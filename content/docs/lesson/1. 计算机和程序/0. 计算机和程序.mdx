# 信息与服务模型

服务侧通过暴露一个通信接口来供客户侧调用能力。

- 操作系统是程序运行的土壤，程序是生长在其上的个体。
- 一切软件、数据、系统，本质上都可以还原为“文件 + 规则”。
- 程序从“源代码 → 机器指令 → 执行”，就是从人类意图到物理动作的转化过程。

计算机世界可以抽象成 **服务与客户** 的关系：

- 服务侧通过接口提供能力
- 客户侧通过调用接口使用能力

- 所有软件、数据、系统，本质都是 **文件 + 规则**。
- 程序的运行就是：**源代码 → 机器指令 → 执行**，即“人类意图 → 物理动作”的转化过程。

# 认识计算机和操作系统

计算机 = 信息处理系统

所有计算机都遵循两层：

1. **硬件**：提供计算与存储能力
2. **软件**：控制、调度、利用硬件能力

> 关键事实：计算机本身只执行“指令序列”，其他一切都是对这一点的扩展。

操作系统是最底层的软件，向软件开发者提供一个调取硬件能力的接口。（操作系统是服务侧，基于操作系统的软件、程序、脚本是客户侧）

所以运行在操作系统上的应用程序又可以叫客户端。

所有的应用程序/操作系统都是文件。（一堆数据）

一个程序从：源代码（人类编写的可读）编译成的二进制代码（机器/硬件可读），既可以在电脑上跑起来。（所有的程序都是）【而编译的过程使用的编译器（可以理解为翻译器），也是一个程序】

所以可以反推，一切程序都是文件。

如果不考虑编码格式： 你可以认为一切的源代码不论后缀是什么，`.py`, `.json`, `.js` 等等，都是人可以读懂的字符串而已。 后缀只是为了让一些编辑器（用户使用编辑器来编写源代码）看懂，并用合适的规则进行提示和高亮。 就像是 `.txt` `.md`



# 从一个脚本开始理解什么是计算机程序

所以理论上，从想法到执行计算机程序，有且仅有三步：

1. 买个电脑
2. 编写代码，保存为一个文件
3. 直接运行

但每个步骤在事实上有都是复杂的，我们可以这么来看。

将一个程序看作是一个主体。 

我们讲一个故事需要什么： 谁，在哪里，做了什么？

who，where，what。

谁就是这个项目。 where是它的环境（context，也是初学者备受折磨的部分），what: 他干了什么活。
